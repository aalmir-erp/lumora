🛠 First-time setup: Deploy the bridge on Railway

The bridge is a separate Node.js service that uses whatsapp-web.js to pair to your personal WhatsApp account (same flow as WhatsApp Web). It must be deployed alongside the bot, with a persistent volume so the auth survives restarts.

One-time cost: Railway free tier covers this comfortably. Bridge uses ~150 MB RAM + 200 MB disk. Total ~$5/month if you exceed free tier.
1

Create a second Railway service in the same project

In Railway → your Servia project → + New → GitHub Repo → pick the same repo → Settings → Root Directory set to whatsapp_bridge.

2

Add a persistent volume (CRITICAL)

Without this, every redeploy logs you out of WhatsApp.

Railway → bridge service → Settings → Volumes → New Volume:

Mount path: /app/.wwebjs_auth
Size:       1 GiB
3

Set environment variables on the bridge

BOT_WEBHOOK=<your-bot-url>/api/wa/webhook
BRIDGE_TOKEN=<long random secret, e.g. click rotate ⟳>
PORT=3001
4

Set environment variables on the bot (this service)

Railway → bot service → Variables → add:

WA_BRIDGE_URL=<bridge-public-url, e.g. https://servia-wa.up.railway.app>
WA_BRIDGE_TOKEN=<same value as BRIDGE_TOKEN above>

After saving, restart both services. The QR appears in this page within ~30s once the bridge is live.

5

Scan + pair

Refresh this page (or click "Refresh QR" above) → scan with WhatsApp on your phone. Once paired, all customer WhatsApp messages flow through the bot.

🩺 Diagnostics

Hit these endpoints to debug:

  • GET /api/health — bot status; wa_bridge: true means WA_BRIDGE_URL is set
  • GET /api/admin/wa-bridge/status — pings the bridge + returns its current state
  • GET /api/admin/wa-bridge/qr — returns the current QR data URL (or "already paired")
